# 电商数据分析智能助手 - 设计规范

## 1. 产品定位

**产品名称：** 电商数据分析智能助手（简称：数据助手）

**目标用户：** 中小电商卖家（淘宝/京东/Shopify），年销售额百万级，无专职数据分析师

**核心价值：** 让没有数据分析能力的电商卖家，轻松读懂自己的生意，并获得行动方向

---

## 2. 核心问题与解决方案

| 现状痛点 | 解决方案 |
|----------|----------|
| 数据分散、口径不一，看不清真实经营状况 | 自动聚合多数据源（DB直连 + 文件导入） |
| 有数据但不会解读，不知道数字好坏意味着什么 | AI 自动解读，输出通俗结论 + 行动建议 |
| 每周手动填 Excel，耗时且数据经常对不上 | 自动化报告，一键生成 PDF/Markdown |
| 知道有问题但不知从哪改，决策靠感觉 | 智能发现异常，主动给出优化方向 |

---

## 3. 产品形态

### 3.1 核心功能

1. **数据接入**
   - 数据库直连（MySQL / PostgreSQL）
   - 文件导入（CSV / Excel / JSON）
   - 统一数据模型，自动标准化电商数据

2. **智能分析**
   - 转化漏斗分析：从浏览→加购→下单的全链路追踪
   - 客户 LTV 分析：RFM 模型，自动划分高/中/低价值人群
   - 同比/环比对比：自动发现异常波动

3. **自然语言查询**
   - 用户用自然语言提问，系统理解意图并返回分析结果
   - 支持多轮对话记忆
   - 例："最近一周加购没下单的用户有哪些？"、"我的漏斗哪一步流失最严重？"

4. **自动化报告**
   - 预置分析模板（日报/周报/月报）
   - 支持 PDF 和 Markdown 格式
   - 一键生成，可定时推送

5. **洞察解读**
   - 分析结果自动配通俗语言解读
   - 主动标记异常点和优化建议
   - 可分享的洞察卡片

### 3.2 用户旅程

```
数据接入 → 系统自动分析 → 查看预置报告
                         ↓
              自然语言提问 → AI 解读返回
                         ↓
              生成可分享报告 → 分享给团队/供应商
```

### 4.3 核心模块职责

| 模块 | 职责 | 关键实现 |
|------|------|----------|
| **Data Source Adapter** | 统一抽象 DB 和 File 数据接入 | Strategy Pattern，支持 MySQL/PostgreSQL + CSV/Excel |
| **Analytics Engine** | 漏斗分析、RFM 计算 | Pandas + SQL 混合计算，大数据分片处理 |
| **NL Understanding** | 自然语言转结构化查询 | LLM API |
| **Insight Generator** | 分析结果转通俗结论和建议 | Prompt Template + LLM |
| **Report Generator** | PDF/Markdown 报告生成 | WeasyPrint + Jinja2 |
| **Security Layer** | 认证、授权、行级权限、审计日志 | JWT + RBAC |
| **Cache Manager** | 查询缓存、计算结果缓存 | Redis，TTL 自动过期 |
| **Task Queue** | 异步大任务 | Celery + Redis |

## 5. 数据模型

### 5.1 核心实体

```
User (用户)
  - id, email, password_hash, role, created_at

DataSource (数据源)
  - id, user_id, type (db/file), config (加密存储), schema_info

Dataset (数据集)
  - id, datasource_id, name, table_name, row_count, last_sync

Conversation (对话)
  - id, user_id, created_at

ConversationMessage (对话消息)
  - id, conversation_id, role (user/assistant/system), content, created_at, metadata

Report (报告)
  - id, user_id, type, content (JSON格式), format (pdf/md), created_at, shared_token

AuditLog (审计日志)
  - id, user_id, action, resource, details, ip, created_at
```

### 5.2 电商标准数据模型（自动映射）

系统预置标准字段映射，支持用户自定义：

| 标准字段 | 说明 | 枚举值/常见来源字段 |
|----------|------|-------------------|
| user_id | 用户标识 | member_id, buyer_id |
| event_type | 事件类型 | **枚举**：impression(浏览), click(点击), add_to_cart(加购), remove_from_cart(取消加购), checkout(结算), purchase(下单), refund(退款) |
| event_time | 事件时间 | create_time, ts |
| product_id | 商品标识 | item_id, sku |
| order_id | 订单标识 | order_no, transaction_id |
| amount | 金额 | payment, revenue |

**event_type 枚举说明：** 系统根据字段值自动识别事件类型，用于漏斗分析。例如字段值为 "add_to_cart" / "加入购物车" / "加购" 均映射到 add_to_cart。

## 6. 安全设计

### 6.1 认证与授权

- JWT Token 认证，支持刷新
- RBAC 角色模型（Admin / Member / Viewer）
- **MVP 简化**：MVP 为单用户体系，RBAC 预留但暂不启用，所有用户默认为 Owner 角色
- 行级数据权限：用户只能看到自己的数据

### 6.2 审计日志

- 所有数据操作记录审计日志
- 登录/登出记录
- 数据导出操作记录
- 支持管理员查看审计日志

### 6.3 数据安全

- 数据库密码加密存储
- 文件上传大小限制
- SQL 参数化查询防注入
- NL 输入验证：用户自然语言查询经过输入清洗和验证，防止 Prompt Injection 攻击
  - 过滤恶意提示词模式
  - 限制输入长度
  - 结构化输出验证

## 7. 商业模式

### 7.1 Freemium 模式

| 层级 | 价格 | 功能 |
|------|------|------|
| **免费版** | ¥0 | 1个数据源、无限分析、AI解读30次/月、报告3份/月 |
| **专业版** | ¥99/月 | 5个数据源、无限报告、AI解读200次/月 |
| **企业版** | ¥399/月 | 无限数据源、无限AI查询、优先级支持 |

### 7.2 定价策略考量

- 免费版足够支撑个人小卖家日常使用
- 专业版覆盖成长型卖家需求
- 企业版面向有团队的卖家
- AI 查询次数限制是核心成本控制手段

### 7.3 留存机制

| 机制 | 频率 | 内容 | 方式 |
|------|------|------|------|
| 周报推送 | 每周一 | 上周数据周报 + AI 智能摘要 | 邮件 + 站内通知 |
| 月报推送 | 每月初 | 上月数据月报 + 趋势总结 | 邮件 + 站内通知 |
| 异常告警 | 实时 | 转化率骤降、流量异常、新增高价值客户 | 站内通知 + 邮件 |

### 7.4 差异化定位

**垂直电商场景专家** — 不做通用 BI，专注电商数据分析：
- 预置电商 event_type（浏览/加购/下单/退款），开箱即用
- 618/双11/平时 三种场景的报告模板
- AI 解读基于电商行业基准（"你的转化率比行业均值低15%"）
- 电商行业特有的异常模式（刷单识别、爆款预警）

## 8. 边界定义

### 8.1 我们做什么

- 数据接入 → 分析 → 洞察 → 报告，一站式
- 零门槛：会说中文就能用
- 智能：主动发现问题，不是被动展示图表
- Freemium：免费可用，付费解锁高级功能

### 8.2 我们不做什么

- ~~专业 BI 工具~~ — 不是给数据分析师用的
- ~~全渠道 ERP~~ — 专注分析，不做进销存
- ~~拖拽式报表设计器~~ — 预置模板为主
- ~~私有化部署~~ — SaaS 模式

## 10. 用户体验设计

### 10.1 设计原则

1. **5分钟价值** — 新用户从注册到看到第一份洞察不超过5分钟
2. **零学习成本** — 会用淘宝/微信就会用，不需要任何培训
3. **始终有AI伴随** — 每个页面都有AI解读，不让用户面对"裸数据"

### 10.2 新用户引导流程

```
注册完成
    ↓
[引导向导 - 拖拽上传]
    ↓
拖拽文件到上传区（支持 CSV/Excel）
    ↓
系统自动识别字段 → 用户确认映射
    ↓
自动分析 → 展示全局概览仪表盘 + AI 总结
    ↓
进入主应用
```

### 10.3 主导航结构（底部 Tab）

```
┌────────────────────────────────────┐
│                                    │
│          [主内容区]                  │
│                                    │
│    根据当前 Tab 展示不同内容          │
│                                    │
└────────────────────────────────────┤
│   [首页]   [报告]   [对话]   [我的]  │ ← 底部 Tab
```

| Tab | 名称 | 内容 | 使用频率 |
|-----|------|------|----------|
| 1 | 首页 | 全局概览仪表盘 + AI 总结 | 每天 |
| 2 | 报告 | 生成的报告列表/新建报告 | 周/月报 |

| 4 | 我的 | 数据源管理/设置/订阅 | 低频 |

## 12. 测试策略

### 12.1 测试金字塔

```
            /\
           /E2E\        ← 10% (3条核心路径)
          /------\
         /集成测试 \    ← 30% (模块协作)
        /----------\
       / 单元测试   \   ← 60% (核心逻辑)
      /--------------\
```

| 层级 | 覆盖内容 | 工具 | 比例 |
|------|----------|------|------|
| **单元测试** | 数据解析、字段映射、漏斗计算、RFM 评分、输入验证、缓存逻辑 | pytest + fixtures | 60% |
| **集成测试** | DB适配器+分析引擎、文件上传+解析、多模块协作 | pytest + testcontainers (PostgreSQL) | 30% |
| **E2E 测试** | 注册→上传→分析、对话查询、报告生成→下载 | Playwright | 10% |

### 12.2 AI 输出验证策略

| 方法 | 说明 | 实现 |
|------|------|------|
| **结构化输出** | LLM 输出必须是 JSON Schema，验证格式 + 必填字段 | Pydantic 模型验证 |
| **黄金集抽检** | 预制20-50个标准问答对，定期跑，监控关键词命中率 | CI 定时任务 |
| **置信度阈值** | 置信度 < 0.6 的回答标记为"不确定"，提示用户 | 输出字段校验 |

### 12.3 数据分析验证策略

```
计算逻辑双重验证
      ↓
Pandas 计算 ──vs── SQL 计算
      ↓
   结果必须一致（允许浮点误差 ±0.001）
      ↓
  不一致则报警（regression）
```

**好处：** 即使未来重构计算逻辑，两套实现结果一致才算正确

### 12.4 数据隔离验证

| 测试场景 | 验证方式 |
|----------|----------|
| 用户 A 的数据 | 查询时强制加 user_id 条件，集成测试覆盖 |
| 数据源权限 | 共享 token 泄露后立即失效 |
| Audit log | 记录所有数据访问，可追溯 |
| SQL 注入 | 参数化查询 + 集成测试覆盖 |

### 12.5 本地 LLM 集成设计

```
┌─────────────────────────────────────┐
│          LLM Config (UI)            │
│  - API 地址（默认支持本地模型）      │
│  - API Key                         │
│  - 模型名称                          │
│  - 温度参数                          │
└────────────────┬────────────────────┘
                 │
┌────────────────▼────────────────────┐
│       LLM Adapter (抽象层)           │
│  - 统一接口                          │
│  - 支持 OpenAI-compatible API       │
│  - 支持 Claude API                  │
│  - 支持本地模型（vLLM/Ollama）       │
└────────────────┬────────────────────┘
                 │
        ┌────────▼────────┐
        │  NL Understanding│
        │  Insight Generator│
        └─────────────────┘
```

**好处：** 开发阶段用本地模型（零成本），生产环境可切换到 Claude/OpenAI，接口不变。

### 12.6 测试数据管理

| 类型 | 管理方式 |
|------|----------|
| 单元测试数据 | fixtures 内嵌，随代码走 |
| 集成测试数据 | testcontainers 启动真实 PostgreSQL，每次测试前重建 |
